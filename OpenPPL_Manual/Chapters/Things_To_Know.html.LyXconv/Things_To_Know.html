Things To Know

If you are a botter who used Shanky-PPL in the past there are some differences you should know about:

No option settings.

Standard Shanky-PPL supports option settings to configure their build-in-default bot; at least that was the old way to do so, but nowadays most of these settings get overwritten by the profile and have no effect at all. As OpenPPL does not support a default bot these option settings have no effect and get ignored.

MakePotSizedPreFlopRaisesWhen = 6 
FoldToPre-FlopRaisesForAQAJsKQ = OFF 
FoldPost-FlopBelowTopPairToAnyBetOrRaise = OFF 
FoldToPost-FlopRaisesWithUnpairedBoardsFor = 2 
...

Cases matter (partially).

OpenHoldem is case-sensitive - contrary to standard PPL. So upper- and lower-cases matter - at least for the translated code. The rules for your OpenPPL-code are more user-friendly and less rigid. There are basically 3 cases:

  • Keywords like “WHEN”, “AND”, “RAISEMAX”, etc. For keywords 3 different spellings are allowed: all upper-cases, all lower-cases and reasonable mixes.
    WHEN (Hand = AA) RAISEMAX FORCE 
    When (Hand = AA) RaiseMax Force 
    when (Hand = AA) raisemax force

    For better readability we recommend to use only upper-cases for keywords.

  • OpenPPL-symbols that are part of the library. They are case-insensitive and you can write them any way you want. The translator knows their correct spelling and is able to generate correct code automatically.
    WHEN (StillToAct < 2) RAISE 3 FORCE 
    WHEN (stilltoact < 2) RAISE 3 FORCE 
    WHEN (sTiLlToAcT < 2) RAISE 3 FORCE

    For better readability we recommend mixed cases like in the library and in this manual.

  • OpenHoldem symbols. Native OpenHoldem symbols are case-sensitive and always lower-cases.
    WHEN (pt_vpip0 < 10) FOLD FORCE 
    WHEN (balance0 < 50) RAISEMAX FORCE

    However: you will probably use these symbols very rarely in your code. Probably only to create poker-logical symbols like PT_OpenRaiser_VPIP and BigBlindStackSize.

OpenPPL needs lots of brackets

Without them the converter worked only 98% of the time. It managed to translate everything, except the most simple cases (of course). In detail:

  • When-conditions need brackets.
    WHEN (StackSize < 10 AND Raises >= 2) RAISEMAX FORCE
  • Hand- and board-expressions need brackets.
    WHEN (Hand = AA) RAISEMAX FORCE 
    WHEN (StackSize < 10 AND (Hand = AK) AND Raises >= 2) RAISEMAX FORCE
  • Not-expressions with equality operators need brackets. In boolean logic the expression
    NOT (a = b)

    is always semantical identical to

    (NOT a) = b

    However there are 2 problems:

    • it is impossible to support the Shanky-semantics of programming here, as a and b can be more complex expressions in OpenPPL and then it would be completely undefined to which sub-expression the not-operator should bind to.
    • if a and b are no boolean values (true / false), but numbers, like in
      NOT StillToAct = 5

      then C++ / OpenPPL would first evaluate “NOT StillToAct” in a boolean context and make every non-zero value false (0) and zero true (1), which is probably not what you want.

    Therefore we decided to stick to “normal” semantical rules and give the not-operator highest priority. You have 2 ways to get what you want:

    • add brackets
      NOT (StillToAct = 5)
    • or use the inequality-operator (that is good maths and that is what Shanky really means with their somewhat crude logic!)
      StillToAct != 5

    We are sorry for the confusion, but we had no other choice but to make it right here.

However: things are not as bad as it may sound: if you have existing code (without the brackets), there is a tool to make it right — bracketify.pl.

  • Prerequisite: ActiveState Perl installed
  • Usage: at the command prompt type:
    perl bracketify.pl < INPUTFILE.TXT > OUTPUTFILE.TXT

    After that you will have a nice profile, with all necessary brackets added and a very nice to read layout. Ready for translation and execution...
