From Maxin Wiki

[edit] Overview

The validator is a new tool introduced with OH 2.0.0, that is aimed to help people to develop more reliable bots and tablemaps. It will execute several hundred consistency-checks at the symbol level to detect invalid game-states, mis-scrapes or incomplete information. The validator will write possible problems to the log-file and also show a message-box, as long as messages are not disabled.

For problems and discussions there exists a validator-forum: [1]

[edit] Preferences

The validator offers several options, available at "Edit -> Preferences -> Validator".

  • Never: the validator will be turned off completely. Not recommended for real-money-play.
  • When it is my turn: the validator will check the rules, when it is your turn, i.e when buttons are visible. This setting offers a good tradeoff between reliability and performance. Recommended for real-money-botting.
  • Always: the validator will check all rules once per heartbeat-cycle. This setting will detect errors, even if it is not your turn, but it causes also a higher CPU-usage. Recommended for verification of new or modified tablemaps and for testing of bots.
  • Use heuristic rules: some rules are heuristic, i.e. they are "common sense", but not always true. Example: A game does usually not last longer than 2 minutes, so the autoplayer has to act at least once per 120 seconds. If it does not do so, there seems to be a problem. Maybe the tablemap does not detect the players cards or the buttons. This setting can result in false positives and is recommended for testing only, but not for real-money-play.
  • Stop autoplayer on error: This setting will quit playing immediatelly, if an error occurs. Recommended for testing, if you want to investigate the error immediatelly, and maybe for very conservative real-money-botting.
  • Shoot replay-frame on error: will shoot a replay-frame in case an error occurs (but of course only once per heartbeat-cycle). This option will enable TM-creators to concentrate on critical frames and reproduce errors more easily. Recommended for both testing and real-money-botting.

[edit] Technical documentation

The validator basically consists of a set of several hundred rules to be checked. Each rules is written in pseudocode, that can be translated into C-macros easily (there is a little Perl-script for that purpose, "rules2cpp.pl"). Each rules consists of

  • a unique ID for the teestcase
  • a flag to distinguish heuristic and non-heuristic rules.
  • some verbal reasoning.
  • a precondition, that checks, if a rule can be applied to the current gamestate.
  • a postcondition, that has to be met, if a rule can be applied.
  • a list of symbols, that got used and are possibly affected in case of an error.

In practice it looks like that:

BEGIN_TESTCASE
    TESTCASE_ID ("0501")
    HEURISTIC_RULE (false)
    REASONING ("If it's my turn, then there has to be always a check or a call button.")
    PRECONDITION ("ismyturn")
    POSTCONDITION (((myturnbits & BUTTON_CHECK) != 0) || ((myturnbits & BUTTON_CALL) != 0))
    SYMBOLS_POSSIBLY_AFFECTED ("ismyturn, myturnbits")
END_TESTCASE
