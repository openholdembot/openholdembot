Optimizing OHF-Files

Profiles coded in Shanky-style often contain long repetitive code-sequences and the converter translates open-ended when-conditions in a not optimal, but easy to translate way. This might lead to larger than necessary code and for large (Shanky-) profiles to somewhat slow execution. To overcome these handicaps our member Relhby developed an optimizer for the generated OH-script-code that replaces repeated code-snippets by auto-generated functions. It gets executed at the command-line like below:

OHF-optimizer.exe INPUTFILE.OHF > OUTPUTFILE.OHF

Advantages

  • The optimized code usually becomes 20-50% smaller, and therefore loads faster.
  • The optimized code usually gets executed 2-10 times faster due to OpenHoldems symbol caching.

Disadvantages

  • For every open-ended when-condition a new function gets introduced with an auto-generated name like “f$OEWC_163”. This affects the readability of your code and especially of your log-files.
  • Well-structured code with lots of user-defined functions and few repetitive code-snippets might even become larger and slower.

General advide

We recommend you to use this tool for original Shanky-style profiles. They do usually benefit much from this optimization. However for well-structured OpenPPL-profiles with lots of user-defined functions the disadvantages might predominate. If you are unsure the file-size before and after optimization will be a good indication. Files that become smaller did usually benefit.
